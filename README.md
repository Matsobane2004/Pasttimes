As third-year software development students at Rosebank College, Lethabo and I set out to build Pastimes, a fully functional, premium second-hand clothing marketplace. We wanted to create a platform that doesn't just look modern and sleek but actually handles real-world e-commerce logic from the ground up. To achieve this, we developed the system using a robust tech stack consisting of PHP for the backend server logic, MySQL for the database architecture, and a custom HTML and CSS frontend to ensure a responsive, contemporary user interface. We specifically aimed for a clean, glassmorphism-inspired aesthetic to elevate the "premium" feel of the second-hand fashion being sold on the site.

The core architecture of Pastimes relies on a relational database structured primarily around users, products, and orders. We built a secure authentication system where new users can register, but Lethabo and I implemented a strict verification gatekeeper to maintain quality control. When someone signs up, their account is flagged as pending in the database. Only administrators—using our hardcoded, secure admin portal—can review and approve these accounts. This prevents spam and ensures our marketplace remains a trusted environment. Once approved, users can securely log in, and their active PHP session begins, allowing them to interact fully with the platform as both buyers and sellers.

For the selling functionality, we engineered a seamless file upload system. When a user lists a piece of clothing, our PHP backend intercepts the uploaded image, renames it with a unique timestamp to prevent file overwrite conflicts, stores it directly in our server's directory, and saves the exact file path to the database alongside the item's brand, size, condition, and price. This data is then instantly queried and rendered dynamically on the homepage and shop sections. We also built a sidebar filtering system so shoppers can efficiently sort through the database by specific categories or premium brands.

Finally, we developed a dynamic shopping cart and checkout pipeline using PHP session variables. Instead of constantly pinging the database every time a user clicks "Add to Cart," our system temporarily holds the specific product IDs in the server's memory, which instantly updates the cart counter in the navigation bar. When the user is ready, they proceed to our secure checkout interface. Upon confirmation, the backend logic loops through their session cart, fetches the exact price data, and injects each item as a permanent row into our orders table—securely linking the buyer's unique ID to the specific product's ID. The session cart is then cleared, completing the transaction. Lethabo and I are incredibly proud of how these interconnected backend systems work together to form a seamless, professional-grade web application.
